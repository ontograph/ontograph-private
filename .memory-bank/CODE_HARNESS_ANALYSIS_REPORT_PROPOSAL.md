# Code Harness Analysis Report Proposal

Date: 2026-06-28

## Purpose

Propose one new repo-owned analysis report shape derived from
`tmp/code-harness` donor ideas.

This is not a new runtime owner and not a new ADR. It is a proposal for a
single output contract that the already-accepted code-harness tool families
could generate later.

## OntoIndex Basis

OntoIndex was used to keep this proposal aligned with the current `codex` repo
and the existing headless-harness decisions.

The unpacked donor package roots under `tmp/code-harness/pypi-src/` are locally
indexed, but donor-root selection through the current OntoIndex MCP and CLI
session was not reliable here. Exact donor report-shape evidence below therefore
comes from direct source reads of the unpacked donor trees.

## Gap

Current memory-bank work already answers:

- what plugin exists
- which tool families are accepted
- which donor ideas are useful

What is still missing is one concrete report artifact that ties those tool
families together in a CI-friendly, headless-friendly way.

Without that, the design still risks drifting into:

- too many small commands with no common summary
- donor-style feature creep
- no stable Markdown or JSON artifact for automation

## Proposal

Add one generated report family:

- Markdown:
  `code-harness-out/ANALYSIS_REPORT.md`
- JSON:
  `code-harness-out/analysis-report.json`

That is enough. Do not add HTML, SARIF, dashboards, sticky-comment logic, or
database-backed history in the first cut.

## Why This Report

This report is the narrow intersection of the strongest donor shapes:

- `code-review-graph`
  - risk-scored changed-scope summary
  - affected-flow summary
  - test-gap summary
  - token-savings metadata
- `cognitx-codegraph`
  - architecture-check JSON artifact
  - audit-report template discipline
  - explicit empty-report-is-valid rule
- `codegraphcontext`
  - cheap complexity and dead-code style local checks

The result should feel like an old-fashioned engineering report, not an agent
chat transcript.

## Proposed Generation Path

The report should be generated by composition of existing accepted tool
families, not by a separate analysis engine.

Proposed pipeline:

1. `harness_status`
   - collect backend authority, freshness, degraded prerequisites
2. optional `harness_update`
   - only when the requested run mode demands refresh
3. `quality_review`
   - compute changed-scope risk, impact expansion, affected surfaces, test gaps
4. `quality_file_check`
   - add cheap local warnings such as complexity, large objects, dead-code
     suspicion
5. `quality_arch_check`
   - add cycles, boundary violations, coupling, orphan checks when requested
6. render
   - write one Markdown report and one JSON report

Skipped:

- separate audit runner
- separate benchmark runner
- separate prompt-driven report generator

Add when:

- a later ADR proves those extra paths are necessary

## Proposed Report Contract

### Frontmatter

Markdown report should start with compact machine-readable metadata:

```md
---
tool: ontocode-code-harness
report: analysis-report
timestamp: <ISO-8601 UTC>
repo: <absolute path>
mode: advisory|strict
scope: staged|unstaged|branch|since-ref|paths
freshness: current|stale-reviewed|updated
backend: ontoindex|other
---
```

Why:

- borrowed from the donor audit-template discipline
- easy to parse
- no need for a separate sidecar header file

### Required Sections

#### 1. Summary

Must include:

- overall verdict:
  `clean`, `review`, or `fail`
- finding counts by severity
- changed files count
- impacted symbols or files count
- exact freshness note
- exact next command when blocked or degraded

This is the top-line CI and human section.

#### 2. Scope

Must include:

- requested scope
- actual analyzed scope
- whether impact expansion widened the review set
- whether arch-check was included
- whether file-check was included

Why:

- donor tools repeatedly distinguish direct change set from expanded blast
  radius

#### 3. Readiness

Must include:

- plugin/backend readiness
- missing prerequisites
- degraded capabilities
- skipped checks and why

Why:

- avoids false certainty when the graph or backend is stale or partial

#### 4. Findings

Each finding should have:

- `id`
- `severity`
- `category`
- `summary`
- `evidence`
- `scope_kind`
  - `direct_change`
  - `impact_only`
  - `repo_policy`
- optional `confidence`

Categories should stay bounded:

- `risk`
- `test_gap`
- `architecture`
- `complexity`
- `dead_code`
- `coupling`
- `orphan`
- `readiness`

Why:

- enough to support both review and CI
- small enough to stay stable

#### 5. Architecture Checks

Only included when requested or when strict mode enables it.

Must summarize:

- cycle violations
- boundary or layer violations
- coupling ceiling hits
- orphan findings
- suppression count

If empty, say so explicitly.

Borrowed directly from the `arch-check` donor family.

#### 6. Test Gaps

Must summarize:

- changed symbols with no direct test evidence
- impacted critical paths with weak test coverage

Keep it short. This is not a full test plan.

#### 7. Efficiency Metadata

Optional in the first implementation, but the contract should reserve space
for:

- saved tokens estimate
- saved percent estimate
- estimate method

Why:

- `code-review-graph` already proves this is a useful compact footer field
- keep optional so it does not block the first slice

#### 8. Skipped Checks

Must list checks not run and why:

- stale backend
- missing data
- unsupported language surface
- strict mode disabled

Why:

- better to report gaps plainly than silently omit them

## Verdict Rules

Keep verdict logic simple:

- `clean`
  - no high-severity findings
  - no blocking readiness issue
- `review`
  - medium or low findings only
  - advisory issues present
- `fail`
  - any blocking readiness issue
  - any high-severity architecture or review finding in strict mode

Do not encode more nuance than that in the first cut.

## JSON Shape

The JSON report should mirror the Markdown sections closely.

Top-level shape:

```json
{
  "meta": {},
  "summary": {},
  "scope": {},
  "readiness": {},
  "findings": [],
  "architecture": {},
  "testGaps": {},
  "efficiency": {},
  "skippedChecks": []
}
```

No nested schema explosion. The report is for automation, not for showing off
schema design.

## Non-Goals

This proposal does not approve:

- SARIF in the first cut
- HTML or dashboard outputs
- comment-updating GitHub actions
- long-lived report history
- watch mode
- database-backed report storage
- agent-written narrative audits

## Recommended First Slice

If this report is ever implemented, start with:

1. Summary
2. Scope
3. Readiness
4. Findings
5. Skipped Checks

Everything else is additive.

That gives one usable report without reopening the whole donor stack.

## Bottom Line

The next useful `tmp/code-harness` derivative is not another tool.

It is one bounded report contract that composes:

- `harness_status`
- `quality_review`
- optional `quality_file_check`
- optional `quality_arch_check`

and writes:

- `code-harness-out/ANALYSIS_REPORT.md`
- `code-harness-out/analysis-report.json`

That is enough to make the headless harness legible to humans and automation
without importing the donors' daemons, installers, or side stacks.
