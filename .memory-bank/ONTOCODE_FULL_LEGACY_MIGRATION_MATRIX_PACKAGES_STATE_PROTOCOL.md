Scope:

- Package, SDK, state/env, protocol, generated schema, telemetry, and MCP/resource/wire identity surfaces.
- Code changes are out of scope.

Default compatibility stance:

- Package identities are release contracts: preserve, dual-publish, or metapackage only.
- Import paths and generated model names are API contracts: preserve or version only.
- `CODEX_*` env/state compatibility is data-loss sensitive: add `ONTOCODE_*` first and continue reading legacy names.
- Telemetry and wire identifiers are downstream contracts: preserve or version only.

## Stage 0 Challenges

- `@openai/codex`, `openai-codex`, and `openai-codex-cli-bin` must not be hard-renamed in the same release as the CLI command cutover.
- `codex_cli_bin` and `openai_codex` are import/runtime contracts and should outlive package download renames.
- `codex-package.json`, `codex-resources`, and `codex-path` are package layout contracts; renaming them needs a layout version and dual-reader support.
- `CODEX_HOME` already has `ONTOCODE_HOME` precedence, so the next work is cleanup and alias hardening, not a first alias implementation.
- Telemetry and app-server generated names require versioning, not local edits to generated files.
- `.codex-plugin`, MCP resource IDs, and protocol method names are integration contracts and should be preserved unless a new versioned protocol is approved.

## Minimum Acceptance Gates Before Implementation

- Run OntoIndex context/impact on every listed target after the current rebuild completes.
- Assign a release/versioning owner for package, protocol, and telemetry rows before code work.
- Decide per package whether `@openai/ontocode`, `openai-ontocode`, and `@openai/ontocode-sdk` are in scope for dual-publish.
- Decide whether schema bundle rename is a new protocol version or permanently out of scope.
- Keep all implementation slices disjoint from `ontocode-rs/` layout rename work.
